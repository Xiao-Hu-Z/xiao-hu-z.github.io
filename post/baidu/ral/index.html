ral

ral

Posted by xiaohu on Tuesday, October 19, 2021

概述

RAL，全称为 Resource Access Layer ，是 ODP（Online Development Platform，在线业务开发平台，面向全百度的在线业务支撑平台） 的资源访问层，以 PHP 扩展的方式提供一个客户端，实现了对后端服务端交互的支持，用户在添加相应服务配置后，即可使用 RAL 一站式接口与后端服务进行交互，而不需要关注数据格式处理与协议交互的过程，为后端交互提供简单可依赖的支持。

特点

  • 支持多种交互协议和数据打包格式 支持 nshead、http发fcgi三种交互协议，string/mcpack1/mcpack2/json/form 五种数据打包协议

  • 性能和简化的追求 高度封装了交互过程，极大地简化上游调用方的开发成本。将 RAL 集成了负载均衡、超时重试、资源定位、配置自动加载等功能，让上游端不需要再关注这些繁琐的通用逻辑

  • 支持资源定位，统一配置，降低运维成本

  • 支持本机配置健康检查和超级负载均衡策略 支持多种均衡策略: 一致哈希、随机、轮询。推荐的负载均衡参数, 可以直接使用配置套餐，负载均衡需要用户以参数形式，指定本次交互的 balance key。

  • 服务自动加载 RAL 支持服务配置自动加载，包括 Local、Galileo、Webfoot Naming Service 三种配置方式，修改配置文件或资源定位的数据发生变化时，RAL 会动态加载最新配置加载时支持数据合法行校验，通过后再更新本地备份。

log日志

ral.log / ral.log.wf 是ral进程的日志文件，查看ral启动、配置更新加载、资源定位成功失败可查看该日志

ral-worker.log / ral-worker.log.wf 是工作进程的配置文件，判读工作进程正常加载及交互成功失败，都可查看该日志

  • logid 唯一标识一次用户请求的id, 如：logid=2526657866，由前端机器生成(小时范围内不能重复)
  • caller 打印该条日志的对象，标识该条日志属于RAL或其他网络交互库, 如: caller=RAL 或 caller=Bd_DB 其他网络交互库使用ral_write_log接口打印日志
  • service 表示请求的服务名称,例如：service_user
  • method表示协议的请求类型，对http包括get、post、put和delete，而对nshead协议来说暂时无意义
  • prot 表示协议类型，如prot=http
  • user_ip 表示本次请求实际用户的ip
  • local_ip 表示本机ip
  • remote_ip 表示本次请求后端服务的ip:port, 如remote_ip=10.10.10.10:8080
  • remote_host(后续支持) 表示请求后端服务的域名,ip存在时优先使用ip
  • uniqid 每次ral调用的唯一uniqid(失败重试的时候会改变)
  • spanid 每次ral请求中的调用id(失败重试的时候spanid不会改变)
  • req_id 每次ral请求的req_id(每个request的req_id都是不一样的)
  • request_name 请求的name
  • reqs_cost 本次ral_multi请求总耗时，ms (一次ral_multi请求中最后一个request的E_SUM日志中)
  • cost 总耗时，ms
  • talk 本次交互耗时，ms
  • connect 链接耗时，ms
  • write 写耗时ms
  • read 读耗时，ms
  • pack 打包的耗时 ms
  • unpack 解包的耗时 ms
  • retry 重试次数 如retry=0/2 表示，第1次交互(0表示未开始重试)，最多2次重试
  • req_start_time 请求开始执行时间(打包之前) ms
  • talk_start_time 请求准备连接时间(打包和负载均衡之后) ms
